ブループリント標準（Bp Standard）v0.1 osakenpiro — 流通フォーマットの先行定義 / 2026-06-12 初版

Ap（アプリ）の後継としての Bp（ブループリント）。
完成品を配布する時代から、雛形を流通させ AI が各ユーザー専用に駆動する時代へ。
アプリが「溶ける」なら、溶けたあとに残って流通するのは Bp だ。

0. 位置づけ

  • Ap→Bp：Application の後継としての Blueprint。流通単位の世代交代。
  • 「溶けるアプリ」＝やりたいことに対して AI がその場でツールを設計する未来（cf. Interop Tokyo 2026 嶋是一氏講演）。Bp はその未来で流通する規格側。
  • 「個別最適の規格化／規格の個別最適化」のハード実装。
  • 外部採用が 0.0 でも、内部統一規格として独立に有用（Float 的保険）。

1. 用語

用語定義
Bp流通単位となる雛形定義。データ構造・様式・AI駆動フックを含む宣言的ドキュメント
駆動（drive）AI が Bp を読み、ユーザー文脈に合わせて実体（ツール・UI・手順）をその場で生成すること
溶けるアプリ都度生成パラダイム。Bp は溶けたあとに残る規格側

2. Bp ドキュメント構造（v0.1 必須5フィールド）

meta:
  id: ...            # 一意ID
  name: ...          # 三重がけ命名（音韻×意味×構造）
  version: 0.1
  author: ...
  license: ...
data:                # データ構造定義（構造は1つ）
  schema: { ... }
views:               # 様式定義（様式は複数）
  - name: ...
    layout: ...
storage:             # 永続化スキーマ
  kind: localStorage # ローカル主権が既定
  key: ...
ai_hooks:            # AI駆動フック＝AIが埋める箇所の宣言
  - slot: ...
    intent: ...      # 何を満たすべきか（実装でなく意図を書く）
    constraints: [ ... ]
    freedom: 0.0-1.0 # AIがどこまで逸脱してよいか（Boolean→Float）

3. 設計原則

  1. データ構造1つ・様式複数 — 同じデータから複数の view を切り出す。
  2. 宣言的 — Bp は「何を」を書く。「どう」は駆動時に AI とユーザー文脈が決める。
  3. ローカル主権 — ユーザーデータは端末側（localStorage 等）。サーバーに送らない。
  4. ゼロコスト互換 — 駆動結果は単一 HTML + GitHub Pages で実体化できること。
  5. Boolean→Float — 駆動の自由度をフィールドごとに 0.0〜1.0 で宣言する。固定すべき箇所と AI に委ねる箇所を、二値でなく連続値で示す。

4. 適用第一陣

Hi-Me TooL 群（KAGAMI / ぼうけんのしょ / HIRAMEKI / HIBIKI / MIERU ほか）を雛形化対象として棚卸し中。いずれも「単一HTML + localStorage + データ構造1つ・様式複数」で実装済みのため、Bp 化の検証母体に適する。

5. 背景ロードマップ（3段）

2胸ポケット録音録画デバイス

6. リスク明示

  • Bp 市場の成立は大手 OS の動向に依存する。ただし外部採用が 0 でも内部統一規格として独立に有用。
  • v0.1 は spec のみ。市場・OS 待ちの段階で実装に過剰投資しない。
